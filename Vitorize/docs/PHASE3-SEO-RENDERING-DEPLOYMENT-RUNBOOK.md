# Phase 3 SEO, rendering, and release runbook

## Release scope

Phase 3 makes anonymous storefront routes server-prerendered while keeping Admin, customer account, authentication, cart, checkout, payment and private operational routes interactive-only and `noindex`. Route metadata is generated by `SeoHead`; robots and chunked sitemap XML are served by the Web host; exact legacy redirects are resolved from `LegacyRedirects` through the API.

No EF migration is used. The only Phase 3 schema upgrade is `Database/Versioned/V0005__seo_content_and_legacy_redirects.sql`.

## Infrastructure prerequisites

- SQL Server 2022-compatible engine/compatibility level tested by the database preflight.
- .NET 8 hosting bundle/runtime for separate API and Web processes.
- HTTPS reverse proxy forwarding the original scheme and host.
- Persistent public media volume for API uploads and a separate private KYC volume.
- External secret provider for SQL credentials (when integrated auth is unavailable), JWT signing key, AES-GCM key, payment/SMS credentials and one-time bootstrap values.
- Backups with a restore test, centralized structured logs, readiness monitoring and alerting.

## Database deployment

1. Back up the target database and record the restore point.
2. Review the immutable manifest and run the deployment runner with `-DryRun`.
3. Execute the runner with an exact `-ConfirmDatabaseName` value. Never execute versioned scripts out of manifest order.
4. Run `Database/Tests/verify_phase2_financial_security.sql` and `Database/Tests/verify_phase3_seo_content.sql` with `sqlcmd -b -f 65001`.
5. Archive the runner log and post-deployment verification output with the release record.

For a fresh environment, publish the reviewed schema-only baseline first. The current baseline remains a candidate and must be approved through the established DBA review; idempotent upgrades reconcile it to the current model.

Rollback is restore-based. V0005 is additive, but do not drop columns/table during an incident: roll application binaries back, disable new redirect rows if necessary, and restore only when data correctness requires it.

## Configuration and secrets

- Configure `ConnectionStrings__DefaultConnection`, `Jwt__SecretKey` and `Encryption__Key` from secrets.
- Configure Web `ApiSettings__BaseUrl` and `ApiSettings__MediaBaseUrl` to the public API origin.
- Set the public setting `Seo.CanonicalBaseUrl` to the single final HTTPS origin, for example `https://vitorize.com`. It must not include a path.
- Choose one host convention at the proxy and redirect the alternate host to it before the application.
- Keep `BootstrapAdmin__Enabled=false` except during the documented one-time bootstrap; remove its secret values immediately afterward.
- Configure SMS.ir and payment sandbox/production values only in their intended environments. Never run E2E fixtures against staging or production.

## Reverse proxy, HTTPS, and headers

Terminate HTTPS at a trusted proxy, forward `X-Forwarded-Proto` and `X-Forwarded-Host`, and allow forwarded headers only from known proxies/networks. Verify HSTS after HTTPS is stable. Preserve the application CSP, frame, MIME-sniffing, referrer and permissions policies. Do not cache authenticated HTML at the proxy.

Production `/robots.txt` allows public content and disallows Admin, account, transactional, search and API paths. Development/Staging deliberately return `Disallow: /`. Verify this distinction after every deployment.

## Canonical, sitemap, and redirects

- Check one home, category, brand, product, blog and static-page response as raw HTML before JavaScript. Each indexable page must have one title, description, self-canonical and appropriate JSON-LD.
- Check `/sitemap.xml` and every referenced `/sitemaps/{kind}-{page}.xml`. Only active/published records may appear.
- Validate reviewed mappings using `Database/Tools/Import-LegacyRedirects.ps1`. Its default mode only validates and generates SQL. Applying requires `-Apply` and exact database confirmation.
- Redirects are exact, queryless source paths. Use 301/308 for replacements and 410 for intentionally removed content. Never add a catch-all redirect to home and never create chains.

## Content migration

Use `tools/content-migration/README.md` and CSV templates. Inventory is read-only; imports require stable external keys, duplicate/slug reports, server-side HTML sanitization, quarantined media validation, alt text, editorial approval and a rehearsal database. Reviews remain unapproved until ownership/rating/date checks pass. Legal content requires owner approval. Keep a pre-import backup and produce external-key-scoped rollback SQL.

## Media and content operations

Admins maintain ProductTags aliases and product mappings in the Product Tags page. Focus keywords are editorial guidance only and are never emitted as meta-keywords. Product/category/brand image alt text must describe the actual image without keyword stuffing. Gallery images use their own alt metadata; decorative imagery uses empty alt. Keep dimensions/aspect ratios and avoid marking non-LCP images high priority.

## Verification commands

```powershell
dotnet clean Vitorize.sln -c Release
dotnet restore Vitorize.sln
dotnet build Vitorize.sln -c Release --no-restore
dotnet test Vitorize.Tests/Vitorize.Tests.csproj -c Release --no-build
powershell -File Database/Tests/Test-DatabaseDeploymentAssets.ps1
sqlcmd -S . -d <disposable-db> -E -C -b -f 65001 -i Database/Tests/verify_phase2_financial_security.sql
sqlcmd -S . -d <disposable-db> -E -C -b -f 65001 -i Database/Tests/verify_phase3_seo_content.sql
```

For browser verification, prepare a disposable database and follow `tests/Vitorize.E2E/README.md`. Preserve Playwright HTML reports/traces. Run `tests/load/public-and-seo.js` with k6 or the PowerShell fallback only against a safe environment.

Also run `dotnet list package --vulnerable --include-transitive`, `npm audit`, `git diff --check`, UTF-8 strict decoding and a secret-pattern scan. Investigate findings rather than suppressing checks.

## Staging dress rehearsal

Use the same SQL major version, HTTPS/reverse-proxy topology, persistent media mounts, external secret provider and realistic data volume as production. Use mock/sandbox payment and test SMS. Rehearse fresh publish and upgrade, backup/restore, two API/Web instances, readiness, outbox lease recovery, restart resilience, cache behavior, file permissions and encryption-key availability. Capture CPU/memory/SQL blocking/connection-pool and p95/p99 latency evidence.

## Post-deploy smoke test

1. Validate API readiness with authorized diagnostics and confirm the public health surface reveals no secrets.
2. Fetch raw HTML, robots, sitemap and one redirect with `curl`/`Invoke-WebRequest`.
3. Complete login, catalog browse, cart, authoritative checkout, sandbox callback, delivery and refund smoke flows with test identities.
4. Verify Admin/customer authorization isolation, private KYC media access and masked sensitive delivery data.
5. Confirm outbox processing, SMS test provider behavior, logs, alerts, backups and retention workers.
6. Monitor 4xx/5xx, payment reconciliation, fulfillment compensation and background-worker errors during the release window.

## Known release gate

Production approval requires the full authenticated Playwright suite and a production-like multi-instance staging rehearsal. The repository harness deliberately does not claim those unexecuted scenarios as passed.
